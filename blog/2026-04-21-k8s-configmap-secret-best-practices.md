Kubernetes 里配置管理看起来很简单：

- 普通配置放 ConfigMap
- 敏感信息放 Secret

但真正上线后，很多事故都不是“不会用”，而是“用得太随意”。

## ConfigMap 适合放什么

适合放：

- 应用开关
- 非敏感环境变量
- 配置文件片段
- 一些可公开的参数

比如：

- 日志级别
- 接口地址
- 功能开关
- 默认超时时间

## Secret 适合放什么

适合放：

- 数据库密码
- Access Token
- API Key
- TLS 证书材料

但要记住：

**Secret 不是“自动绝对安全”。**

在很多默认环境里，它只是 base64 编码存储，不等于天然强加密。

## 最常见的坑

### 1. 把所有配置都塞环境变量

环境变量很方便，但也有问题：

- 更新不够直观
- 配置项一多会很乱
- 某些场景下排查不够友好

配置复杂时，用挂载文件通常更清晰。

### 2. Secret 跟源码一起飞

最危险的情况，是把包含明文敏感值的 YAML 直接提交到 Git 仓库。

这类事故非常常见，而且一旦进了 Git 历史，就不只是“删掉文件”这么简单。

### 3. 配置变更没有发布策略

有些应用改了 ConfigMap，但 Pod 不会自动重启。

结果就是：

- 你以为改了
- 实际线上没生效
- 排查半天怀疑人生

所以配置变更要么配合 rollout，要么引入更明确的 reload 机制。

## 一个比较稳的习惯

我更建议这样分层：

- 默认配置：写进应用默认值
- 环境差异配置：ConfigMap
- 敏感配置：Secret
- 更高敏感度密钥：外部密钥系统（比如 Vault / 云 KMS）

## 判断标准其实很简单

你可以问自己两句：

1. 这东西泄露了会不会出事？
2. 这东西是不是应该跟代码版本解耦？

如果第一句答案是“会”，优先按 Secret 或外部密钥方案处理。
如果第二句答案是“是”，就别把它写死在镜像里。

## 一句话总结

ConfigMap 和 Secret 真正重要的，不是“会创建”，而是你有没有把配置、密钥、发布流程三件事拆清楚。
