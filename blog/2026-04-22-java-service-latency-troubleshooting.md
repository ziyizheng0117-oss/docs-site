线上 Java 服务 RT 飙高，是很典型但也很容易误判的问题。

很多人一上来就看 CPU，或者直接怀疑数据库。但真正排障时，最重要的不是先猜答案，而是先判断：**慢，究竟慢在应用内部，还是慢在外部依赖。**

## 我一般先看三件事

### 1. 整体 RT 变化趋势

先确认是：

- 所有接口都慢
- 只有少数接口慢
- P99 飙高，但平均值变化不大
- RT 变高的同时，错误率有没有一起升

这一步的意义是判断问题范围。

如果只有某几个接口慢，通常优先怀疑：

- SQL 变慢
- 下游 RPC 抖动
- 某个新逻辑引入了阻塞

如果所有接口一起慢，就要怀疑更底层的共享资源：

- 线程池
- 连接池
- GC
- 网络
- 宿主机负载

### 2. 线程池和连接池有没有被打满

Java 服务 RT 飙高，最常见的根因之一不是“代码执行慢”，而是**排队时间变长**。

比如：

- Web 容器线程池满了
- 业务线程池积压
- 数据库连接池被借空
- HTTP 客户端连接池耗尽

这类问题的典型特征是：

- CPU 不一定高
- GC 不一定异常
- 但请求会越来越慢
- 最后开始超时

### 3. GC 有没有突然恶化

如果是新版本上线后 RT 抖动，GC 一定要看。

重点不是只看一次 Full GC，而是看：

- Young GC 频率是不是异常变高
- 单次停顿时间是不是变长
- Old 区增长是不是明显加快
- 是不是有对象分配速率突然变大

如果 RT 呈现周期性尖刺，GC 的嫌疑会非常大。

## 然后我会继续拆内部和外部

### 应用内部慢的典型线索

如果你看到这些现象，通常更像应用内部问题：

- CPU 持续偏高
- 某类线程长期 RUNNABLE
- 锁竞争明显
- GC 时间显著上升
- 某次发布后立即出现

这时一般要继续看：

- 火焰图
- 线程栈
- 锁等待
- 热点方法
- 代码路径是否引入了同步阻塞

### 外部依赖慢的典型线索

如果这些指标更明显，就优先查下游：

- 数据库慢查询明显增多
- Redis 响应抖动
- 某个 RPC 调用耗时明显变长
- 网络重传或 DNS 异常
- 上游重试把流量放大

很多时候，应用 RT 飙高只是表象，真正慢的是它依赖的那个东西。

## 很多人会忽略的一点：看“变化点”

排障效率最高的方法之一，就是找最近的变化：

- 有没有刚发版
- 有没有改线程池参数
- 有没有切流量
- 有没有换数据库索引
- 有没有新任务批量跑起来
- 有没有节点漂移或宿主机异常

没有变化的系统当然也会出问题，但现实里，**大部分线上故障都和某个变化点强相关。**

## 一个很实用的排查顺序

我自己的顺序通常是：

1. 看 RT、QPS、错误率的关联
2. 看线程池 / 连接池 / 队列长度
3. 看 GC 和堆内存趋势
4. 看下游依赖耗时拆分
5. 看线程栈、热点方法、锁竞争
6. 回到变更记录确认触发点

## 一句话总结

Java 服务 RT 飙高时，最怕的是一开始就陷入“拍脑袋猜根因”。

先判断是排队、计算、GC 还是下游依赖，再往下钻，通常会比“盲看 CPU 和日志”快得多。
