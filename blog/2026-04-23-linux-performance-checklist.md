Linux 机器变慢这件事，说简单也简单，说容易乱也是真的容易乱。

最常见的问题不是“不会查”，而是信息一多就开始东看一点、西看一点，最后看了半天还是不知道瓶颈在哪。

所以我更喜欢按固定顺序排查。

## 第一步：先分清楚是“系统慢”还是“应用慢”

先问自己两个问题：

1. 是整台机器都慢，还是某个服务慢？
2. 是持续慢，还是偶发抖动？

如果整台机器都卡，优先看系统层：

- CPU
- 内存
- IO
- 网络
- load average

如果只有某个应用慢，那就不要一开始把所有锅都甩给 Linux，本质上还是要回到应用自己的线程、连接池、依赖和日志。

## 第二步：看 CPU，不要只看使用率

很多人只盯着 CPU usage，但更关键的是：

- user 高还是 system 高
- 有没有单核打满
- 有没有大量上下文切换
- 有没有异常软中断

比如：

- user 高，可能是应用本身在算
- system 高，可能是内核态压力、网络、IO 或锁竞争
- iowait 高，说明 CPU 不是忙，而是在等 IO

## 第三步：看内存，重点不是“剩余多少”

Linux 的内存看法和很多人直觉不一样。

重点要看：

- available memory
- swap 是否开始使用
- page cache 是否被频繁回收
- OOM 是否出现过

如果 swap 开始明显参与，而你的应用又是低延迟服务，那体验通常会很差。

## 第四步：看 IO，很多“慢”其实是在等磁盘

典型表现：

- CPU 不高
- 负载却很高
- 请求一个个超时
- 磁盘 util 接近 100%
- await 很高

这时候不是“程序算不过来”，而是整个系统在排队等 IO。

## 第五步：看网络，尤其是云环境

在云主机或容器环境里，网络问题非常常见：

- 丢包
- 重传
- DNS 慢
- 某个下游连接建不起来
- NAT 或出口带宽打满

很多人排障时只看本机指标，但真正拖慢服务的可能是网络链路上的波动。

## 第六步：结合变化点，不要脱离时间线

和所有线上问题一样，Linux 性能问题也很怕脱离时间线。

你最好同步确认：

- 最近有没有发布
- 有没有跑新批任务
- 有没有切换实例规格
- 有没有容器迁移
- 有没有新挂载磁盘或日志暴涨

指标只告诉你“现在发生了什么”，变更记录才更像“为什么从这一刻开始发生”。

## 一个够用的排查框架

我更喜欢把 Linux 排障压缩成五个字：

**算、存、盘、网、变。**

- 算：CPU、调度、上下文切换
- 存：内存、swap、缓存回收
- 盘：IO 吞吐、延迟、队列
- 网：延迟、丢包、重传、DNS
- 变：最近有没有改东西

## 一句话总结

Linux 机器变慢时，不要一上来就到处抓日志。

先判断是 CPU、内存、IO、网络还是变更带来的问题，再往下拆，排障路径会清楚很多。
